# Visible Increments — Sprint 7 Commit-by-Commit Log

**Purpose (CTO direction, 2026-07-18 — the "Visible Increment Rule"):** a running visual timeline of Sprint 7, so the sprint leaves both a clean Git history and a chronology of Book Publisher Studio actually coming into existence. This is distinct from `docs/demo/screenshots/`, which is the curated, final 6-image Demo Script capture set produced once at Commit 11 — this log is incremental, produced at every commit along the way, and never retroactively cleaned up or replaced.

**This file is the single source of truth for Sprint 7's visual history (CTO direction, 2026-07-18).** An earlier proposal to also add a separate `docs/demo/timeline/` directory (one file per commit) plus a manually-maintained `Sprint7Timeline.md` was reconsidered and rejected in favor of one pipeline: this file accumulates every entry during the sprint; at Sprint 7 closure (Commit 12), it is compiled into a polished `docs/releases/v0.8.0-alpha/SPRINT_7_TIMELINE.md` (matching this project's existing `docs/releases/<version>/` convention for sprint-closure artifacts), with an optional PDF export derived from that Markdown if useful at that point. No duplicated tracking structure exists during the sprint itself — this file is it.

**Scope extended (CTO direction, 2026-07-18, after Commit 3):** originally scoped to `frontend/`-touching commits only. Now covers every implementation commit from Commit 3 onward, with two artifact types depending on what the commit actually changed:

- **Backend/tooling commits** (no UI yet to screenshot) — a small conceptual diagram (SVG, two or three labeled boxes + arrows) showing the architectural step in one glance, e.g. "Workspace → Shared DTOs."
- **`frontend/`-touching commits** — a real screenshot of the running dev server, per the original rule (see below) — never a mockup.

Both get the same accompanying **"what's now true/usable"** description and **"confirmed real, not simulated"** statement.

## What each entry contains

Per commit, appended below (never edited retroactively — this is a timeline, not a status page):

1. **Artifact** — a real diagram (backend/tooling commit) or a real screenshot of the running `frontend/` (UI commit), stored in `docs/demo/visible-increments/commit-NN-<slug>.{svg,png}`, embedded inline.
2. **What's now true/usable** — 1-3 sentences, plain language, describing the concrete new capability or architectural state a person could verify by hand (not "types added" — see `docs/DEVELOPMENT_WORKFLOW.md`'s Visible Increment Rule, which this log is the evidence trail for).
3. **Confirmed real, not simulated** — explicit evidence, not an assertion: the real HTTP request/response observed (curled directly or via the Browser pane's network inspector), the real fixture file used, or the real test/build output — quoted directly.

### Commit 9a — `feat(frontend): PDF preview`

**Real screenshots captured** — the Preview section with its real, dimmed "stale" state and the amber staleness warning after a layout change (see below), matching the earlier Commit 8 screenshot success.

**CTO-directed split, this session:** the originally-planned single Commit 9 (`export + preview`) was split into two verified sub-steps at the CTO's own request — 9a (preview only) and 9b (export/download) — so that if a problem appears in the export pipeline, it's immediately clear whether it's the document generation or the download mechanism. Both remain "Commit 9" in the Design Review's own numbering; Commit 10 (real-file verification) is unaffected.

**What's now true:** the format/layout selector (Commit 8) is now followed by a real PDF preview. `PreviewPanel` (new component, fourth sibling under `UploadDropzone`'s success branch) fires a real `POST /api/manuscripts/export?format=pdf` for the file that was actually dropped, using the currently selected layout/theme, when "Generate Preview" is clicked (Decision 1: full re-export, not an incremental preview system) — no request fires automatically on selection change. The real returned PDF is embedded (`<embed type="application/pdf">`) and a real page count ("≈ N page(s)") is derived from the same PDF's actual bytes (a `/MediaBox` occurrence count — the same heuristic `backend/src/test-utils/extractPdfText.ts` already uses for this project's own PDFKit output, not a new invented technique). Changing the layout or theme after a preview exists is detected at render time (not via a state-resetting effect) and shows a real "stale, click Regenerate" warning with the old preview dimmed, rather than silently showing an out-of-date PDF as if it were current.

**Explicitly not built, by CTO-confirmed design, not an oversight:** a live page estimate per layout option inside the selector itself (raised as part of the same conversation as this commit). That would need one export call per layout before the user has chosen anything — six speculative round trips against a stateless backend built for one-round-trip-per-action (Sprint 7 Decision 2). The real number shown here is for the one, currently-selected layout, generated by one real user-initiated action, not fabricated and not fetched speculatively.

**Real bug caught by ESLint's `react-hooks/set-state-in-effect` rule before any browser check ran:** the first draft reset preview state via `setState` inside a `useEffect` keyed on `layout`/`theme`, which the linter correctly flagged as a cascading-render anti-pattern (it would have fired on mount too, not just on genuine changes). Redesigned to compute staleness as a derived value during render instead, with a separate cleanup-only effect (no `setState` in its body) solely for revoking the blob URL on unmount.

**Disclosed tooling incident, more severe than prior recurrences — not a product defect:** the frontend dev server exhibited persistent `global-error.js` React Client Manifest errors (the same class first logged at Commit 4) that this time did **not** self-resolve on retry or even a full process restart — roughly half of all requests kept 500ing across two separate `preview_start` cycles, with new companion errors (`Manifest file is empty`, `components.ComponentMod.handler is not a function`, `SyntaxError: Unexpected end of JSON input`) appearing alongside it. Diagnosed via `preview_logs` rather than guessed. Resolved by clearing the stale Turbopack dev cache (`rm -rf frontend/.next`) and restarting — five consecutive clean `200`s followed, and the rest of this commit's verification ran without recurrence. This is the same CTO-tracked "watch point" from Commit 4, now escalated in the record as materially worse than earlier occurrences — a `.next` cache clear is the working remedy if it recurs again, not just a retry.

**Confirmed real, not simulated:**
