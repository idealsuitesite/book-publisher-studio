# Render Metrics — Level 2 Design Review

**Status:** ✅ **APPROVED** — CTO, 2026-07-18, round 1, without modification. Implementation authorized.
**Date:** 2026-07-18
**Trigger:** ADR-0038, deferred OPEN since Sprint 8, which framed one question and deliberately left it unanswered:

> How should `LayoutEngine`'s computed metrics be exposed to the Publishing Engine without breaking the existing separation of responsibilities?

This review answers it. It also records a **product defect found while writing it** (§3) that ADR-0038 does not mention and that no test currently catches.

---

## 1. Objectives

Close the gap where the publish pipeline computes real pagination, then discards it before the component that needs it can see it. Three consumers are blocked by this, not one:

| Consumer | Needs | State today |
|---|---|---|
| `PageCountRule` | real page count vs KDP's 24–828 | `PAGE_COUNT_UNKNOWN` on every real fixture |
| Paperback cover spec | `spineWidthInPerPage × pages` | **inconstructible** — no rule exists yet |
| KDP gutter margins | `marginsByPageCount` lookup | **inconstructible, and see §3** |

ADR-0038 describes only the first. The other two were found by reading `KDPRuleData.ts` for this review.

---

## 2. Current state — evidence, not assumptions

Every claim below was confirmed by reading the source, per `DESIGN_REVIEW_PROCESS.md` §2.

**The metrics exist and are then dropped.** `PublishingUseCase.execute()` ([PublishingUseCase.ts:46](../../../backend/src/application/use-cases/PublishingUseCase.ts#L46)):

```ts
const paginated = this.layoutEngine.paginate(typeset, request.pageLayout);
const pdf = await this.renderer.render(paginated, { language: book.metadata.language });
return this.publishingTarget.prepare(book, { pdf });   // ← `paginated` dies here
```

`paginated.pages` is the real paginated result. `prepare()` receives `book` and bytes.

**The two page counts are different quantities and must not be merged.** `Book.pageCount` is populated only by `BookMetricsCalculator` on the **import** path, as `estimatePageCount(wordCount)` — an estimate from word count. `PaginatedBook.pages.length` is the real typeset result. They will not agree.

**`PageCountRule` is correct and needs no behavioural change** ([PageCountRule.ts:15](../../../backend/src/domain/services/publishing/PageCountRule.ts#L15)) — it reports `PAGE_COUNT_UNKNOWN` honestly rather than guessing. It is being fed less than the pipeline knows.

**`RenderedOutputs` is bytes only** — `{ pdf?, epub?, docx? }`.

## 3. Defect found while writing this review: KDP gutter margins do not exist

**This is not a transport problem. It is a missing feature that makes every paperback non-compliant.**

`KDPRuleData` records KDP's real requirement — the inside (gutter) margin scales with page count:

| Pages | Gutter required |
|---|---|
| ≤150 | 0.375 in |
| ≤300 | 0.5 in |
| ≤500 | 0.625 in |
| ≤700 | 0.75 in |
| ≤828 | 0.875 in |

`PageLayout` ([PageLayout.ts:1](../../../backend/src/domain/models/PageLayout.ts#L1)) offers `marginTop/Bottom/Left/Right` and **nothing else**. Confirmed by grep: the string `gutter` appears in neither `PageLayout.ts` nor `LayoutEngine.ts`.

Consequences, stated plainly:

- Margins are **symmetric**, so there is no inside/outside distinction and no recto/verso notion at all. Every page is laid out as if it were a loose sheet.
- A 400-page book generated by this product today has a 0.625 in gutter requirement and gets whatever `marginLeft` happened to be. Text runs into the binding.
- **828 pages of tests do not catch this**, because no test asserts a requirement the model cannot express.

**There is also a genuine circularity here, and it should be named rather than discovered mid-implementation:** the required gutter depends on page count, and increasing the gutter reduces text per page, which increases page count. A book near a threshold (150, 300, 500, 700) can cross it *because* of the margin its own page count demanded.

**Recommendation: fix the gutter separately from this review, and do the metrics work first.** The gutter fix needs `PageLayout` to gain a gutter/inside-outside concept and `LayoutEngine` to honour it — that is a rendering-pipeline change with visual consequences for every existing export and its own baseline implications. Folding it into a metrics-transport change would produce a commit that cannot be reviewed. It should get its own ADR and its own review.

For the circularity, the recommended resolution when that review happens is **one re-pagination pass, not convergence to a fixed point**: paginate, look up the gutter for the resulting count, re-paginate if the gutter changed, and validate the result. If the second pass crosses another threshold, report it as an issue rather than looping — an author whose book sits exactly on a boundary needs to be told, not silently given a third layout.

## 4. Proposed decisions

### Decision 1 — A new `RenderMetrics` Domain model, narrow and platform-agnostic

```ts
export interface RenderMetrics {
  pageCount?: number;        // real, paginated. Absent for reflowable formats.
  pageLayout: PageLayout;    // the geometry pagination actually used
}
```

Platform-agnostic per ADR-0037: it holds facts about a rendition, no platform name, no KDP concept. `pageCount` is **optional and that is load-bearing** — an EPUB is reflowable and genuinely has no page count, so `PAGE_COUNT_UNKNOWN` keeps a legitimate life. It stops firing on every PDF, which is the actual fix.

### Decision 2 — Metrics travel per format, alongside the bytes that they describe

```ts
export interface RenderedOutput { bytes: Buffer; metrics: RenderMetrics; }
export interface RenderedOutputs { pdf?: RenderedOutput; epub?: RenderedOutput; docx?: RenderedOutput; }
```

A bundle may hold a paginated PDF and a reflowable EPUB at once. One page count for the bundle would have to be wrong about one of them. Attaching metrics to the artifact they describe is the only shape that stays true for both.

### Decision 3 — `PostRenderValidationContext` carries the metrics; rules read them there

```ts
export interface PostRenderValidationContext {
  book: Book;
  bundle: PublishingBundle;
  metrics: Partial<Record<'pdf' | 'epub' | 'docx', RenderMetrics>>;
}
```

`PageCountRule` gains a constructor argument naming which artifact it validates, consistent with every other rule taking its config by constructor so a future `KoboRuleProvider` can reuse it (Sprint 8 Decision 7).

### Decision 4 — The estimate and the real count never merge, and the rule never falls back

`Book.pageCount` stays the import-time estimate, untouched. `RenderMetrics.pageCount` is the real one. **`PageCountRule` must not fall back to `Book.pageCount` when metrics are absent.** KDP rejects on the real count; validating an estimate against a hard limit would turn a `WARNING` into a `PASS` on a book Amazon will refuse. A false green is worse than the honest unknown we ship today.

Writing the real count *into* `Book.pageCount` — ADR-0038's candidate 4 — is rejected for the same reason plus a second: the same field would mean two things depending on which path produced it, and no consumer could tell which it held. A page count is a property of a rendition, not of the work (`AGGREGATES_AND_PERSISTENCE.md` §1).

### Decision 5 — `PaginatedBook` does not cross into publishing

ADR-0038's candidate 3 is rejected. `PaginatedBook` carries `styledBook`, `pages`, `pageLayout` and the TOC — the whole content. Publishing needs counts, not content. Passing it would couple the publishing port to the layout engine's internal model, so every layout change would ripple into publishing.

### Decision 6 — `prepare()`'s widened signature is a planned evolution, not a break

Sprint 8 Decision 1 recorded in the port itself that `prepare()` was "Sprint 8's only operation, not necessarily this port's final API". This is that anticipated evolution. `PublishingTarget` has exactly one implementation (`KDPTarget`), so the change is contained.

## 5. Open questions — for CTO decision

**Question 1 — does `Renderer` produce the metrics, or does `PublishingUseCase` assemble them?**

> ⛔ **This recommendation was approved and was wrong. Corrected by ADR-0045 during implementation.** Kept as written, per ADR-0010's annotate-never-rewrite precedent.
>
> It was wrong twice over. `PaginatedBook.pages.length` is an *estimate* that ADR-0013 already recorded as drifting from the real rendered count — `PDFRenderer` says so in its own comments and uses `bufferedPageRange().count` instead for exactly that reason. And renderers emit title and copyright pages that pagination never sees. On the canonical fixtures the reported count was **1** while the shipped PDF had **3**.
>
> The real answer: **the renderer produces the metrics**, because only the renderer knows what it emitted. `Renderer<TOutput>` now returns `RenderResult<TOutput>`, and ADR-0012 does change after all. Cost: 55 call sites, nearly all test assertions.
>
> The reasoning error is worth naming: this question was settled by weighing blast radius instead of reading `PDFRenderer`, which contained the fact that falsified it. "Confirmed, not guessed" (ADR-0031/0032) applies to our own code, and approval did not make the answer true.

*Original recommendation: `PublishingUseCase` assembles them.* It already holds `paginated`; the `Renderer<TOutput>` port (ADR-0012) has three implementations and widening its return type changes all three plus the export path, for no gain. **This keeps ADR-0012 untouched.**

**Question 2 — does the export path get metrics too?**
*Recommendation: not in this scope.* `ExportManuscriptUseCase` has no validator to feed. Adding metrics there would be unused code, and §7 of this project's own handbook rule (port when more than one real implementation is plausible) applies equally to fields.

**Question 3 — should the paperback cover spec (spine width) be built in this scope?**
*Recommendation: no — metrics first, cover rule second.* This review unblocks it; building both at once means the cover rule and its transport land in the same commit and neither can be reviewed independently. It should follow immediately after, in the same sprint.

## 6. Risks

1. **The real count may reveal that fixtures fail KDP's 24-page minimum.** `typography-test.docx` is small. Expect real `PAGE_COUNT_OUT_OF_RANGE` errors to appear where a comfortable `WARNING` sat before. That is the fix working, not a regression — but it will change `verify-real-publish` output and must not be "fixed" by loosening the rule.
2. **Gutter (§3) stays broken after this work.** This review makes the defect *visible and validatable*; it does not make the product compliant. That must be stated in the sprint's release notes rather than implied by a green publish report.
3. **Metrics are only as correct as `LayoutEngine`.** Transporting a wrong page count faithfully is still wrong. Real-fixture verification (`npm run verify-real-publish`) is mandatory here — this project has found five real bugs that passed full synthetic coverage (ADR-0019, 0020, 0031, 0032, 0038).
