- **`HEURISTIC_STRUCTURE_DETECTION` — CLOSED ON EVIDENCE, not paused for want of an idea** (2026-07-21). Detecting chapters in a DOCX whose author applied no Word heading styles is **out of reach with the material the import pipeline can obtain** — not "we have not found the right algorithm yet". **Nobody should reopen this believing a better heuristic would solve it.**

  **The structural reason, which is what makes this closed rather than open:** the properties that survive to us are precisely the ones a real author never uses to mark a heading. Word stores heading formatting in the paragraph *style*; mammoth reports only *directly-set* properties. So an author who marks headings properly makes them invisible to every signal we can read, and an author who bolds them by hand drowns them among hundreds of other bold paragraphs.

  **Measured in three steps, each closing an avenue at the cost of a spike rather than a chantier** (`backend/spikes/heuristic-signals-spike.ts`, `mammoth-capability-spike.ts`, `ooxml-inheritance-spike.ts`, `heuristic-signals2-spike.ts` — on branch `spike/heuristic-structure`, unmerged):
  1. **HTML conversion discards formatting** — surviving attributes: NONE. But `transformDocument` *does* expose `fontSize`, `isBold`, `isAllCaps`, `alignment` on the underlying document objects, with no new library and no port change.
  2. **`fontSize` is populated only where it is not needed** — 725/836 runs on the already-styled book, **0 of 2698 / 0 of 141 / 0 of 30** on the unstyled ones. Resolving the full OOXML inheritance chain (`docDefaults` → style → direct) yields **12pt for every one of 2660 paragraphs**: the size is absent from the source, not merely dropped by mammoth. **The raw-OOXML avenue is therefore closed too, and was closed before any integration was costed.**
  3. **`alignment` and `isAllCaps` are empty on all four fixtures.** Against the only ground truth available (faith-alone, 17 real `Heading 1`), the best candidate rule scores **precision 0.0%, recall 0.0%** — it finds 16 paragraphs, not one of which is a real title.

  **CTO threshold, fixed BEFORE the measurement so it could not move to fit the result:** under 5% false positives would have been usable *as a suggestion, never as silent truth* (the ADR-0049 explorable-finding pattern — a false chapter accepted silently is worse than an honest "0 chapters detected", because the first invites misplaced trust and the second invites correction). The result is nowhere near it.

  **Still open, deliberately unmeasured:** the *semantic* half (quotes, dialogue). It probably suffers the same mechanism — `<blockquote>` is already known to appear only when the author applied Word's Quote style (`C1_QUOTE_PRESENTATION_UNBLOCK`) — but **probably is not measured**, and this chantier demonstrated three times why that distinction is what stops a session wasting itself on an unverified conclusion. Not prioritised.

- **`DROPCAP_TEXT_OVERLAP`** — **real rendering defect, PDF only — FIXED and shipped in PR #26** (found 2026-07-21, CTO-classified; fixed in the same window). Named independently of the Book Presentation chantier: this is a text-layout bug that the theme work merely uncovered, not a theming question. **The measured facts below are preserved as the record of what the defect WAS**; the fix indents the spanned lines beside the glyph and prices the resulting extra height in the model (`LayoutEngine.priceDropCapParagraph`, mirroring `PDFRenderer` step for step), locked by `dropCapOverlap.test.ts`.

  **Measured proof.** Source text: `"Every paragraph in this measurement carries the same words…"`. Rendered PDF output, visually verified at 100% in a real viewer: line 1 reads `"Every paragraph in this measurement"`, line 2 reads `"ies the same words so that the only"` — **the characters `carr` are painted over by the drop-cap glyph and unreadable.** Four characters lost on every drop-cap paragraph, at exactly the spot chosen for visual emphasis. Reproduce with `npx tsx backend/spikes/dropcap-render-pdf.ts`.

  **Cause, located.** `PDFRenderer.renderRunsWithDropCap` draws the glyph at 2.5× via `doc.text(char, { continued: true })` and reserves **no horizontal indentation** for the lines the glyph descends into. **The defect is horizontal, not vertical** — reserving vertical space would not fix it, since the glyph would still cover the start of line 2. The minimal real fix is indenting the spanned lines (a reduced horizontal wrap, not a full wrap-around engine).

  **PDF only — verified in all three renderers, not assumed.** EPUB uses `.dropcap { float: left }`, so the reading system's own CSS engine performs a real text wrap: correct, no overlap. DOCX emits an enlarged `TextRun` inline, and Word gives a line the height of its tallest run, so the first line grows instead of characters being hidden: degraded (a big first letter, not a true drop cap) but **lossless**. *(EPUB read in code and in its CSS; DOCX read in code and reasoned from Word's line-box behaviour — not opened in Word.)* Worth noting on its own: three renderers, three different drop-cap behaviours — proper, degraded, and corrupting.

  **It was latent when found — and is now fixed regardless.** Nothing populates `Block.dropCap` anywhere in the pipeline (measured: 0 drop caps across 2,152 real paragraphs, `dropcap-feasibility-spike.ts`), so the defect never had a producer and no shipped book ever suffered it. That made the fix urgent-free, not optional: it was corrected in PR #26 *before* any producer could exist, so building the positional trigger later would hand a real user a **correct, priced** drop cap — not a latent bug. The drop-cap **theme capability** stays frozen for a separate reason (three renderers, three drop-cap strategies — see `MINI_DR_DROP_CAPS.md` §4bis), not because this defect is still open.

  **NOT counted in the eleven-bug real-fixture lineage** (CTO ruling). It was found by targeted instrumentation on a synthetic 120-paragraph fixture, not by a real corpus manuscript. The lineage counter measures specifically what *real files reveal that synthetic coverage misses*; counting this would invert its meaning. Recorded as a real bug found by deliberate instrumentation, a separate category. See `docs/REAL_FIXTURE_POLICY.md`.

  **Pricing was reinstated and shipped — the earlier removal is annulled.** The 2026-07-21 decision removing `estimateBlockHeight` pricing for `dropCap` rested on a measured height invariance (25 → 25 pages, 0 → 0 reconciliations) that Word's own layout later showed to be **the signature of this very bug**, not evidence of no cost: Word goes 3 → 4 pages on the same 61 lines and 397 words, because it grows line boxes where PDFKit overlaps them. Annulled formally, with the full reasoning, in `docs/DECISIONS.md` — *"R2 verifies that charged equals consumed, not that what is consumed is correct"*. Pricing was the **central** subject of the fix and shipped **with** it: `LayoutEngine.priceDropCapParagraph` charges the extra lines the narrowed-column indent creates, and the reversal now lives on `main`.

  **The mini Design Review it needed was done and implemented.** `MINI_DR_DROPCAP_OVERLAP.md` (CTO-approved) answered how R2 stays guaranteed once horizontal indentation enters the picture; the fix landed in PR #26 (`4c75879`). What remains gated is the drop-cap **theme capability** (the positional trigger + theme/renderer wiring), and for a different reason than this defect — see `MINI_DR_DROP_CAPS.md` §4bis.

- **`DROPCAP_PARAGRAPH_ATOMICITY`** — accepted disclosed tech debt (CTO decision 2026-07-21, same status as its cousin `LIST_SPLITTING_ACROSS_PAGES` above). A paragraph carrying a drop cap is excluded from Phase B line-splitting (`LayoutEngine.ts:222`, deliberate — a drop cap's first lines are typographically special), so a drop-cap paragraph that does not fit the remaining page space cannot split: it moves whole and, if the page cannot hold it, produces a reconciliation page. **Not corrected now, and correcting it is not planned.**

  **The justification is STRUCTURAL, not about the drop cap's height.** The exclusion makes the *whole paragraph* atomic regardless of what the glyph costs; a long drop-cap paragraph overflows for exactly the reason a long list item does — **it cannot be split, full stop**. This is independent of pricing, and deliberately so: the debt survives whatever the drop cap costs. **(Corrected 2026-07-21 — do not restore the earlier wording.)** An earlier version of this entry argued the exclusion was harmless because "a drop cap consumes zero extra height (25 → 25 pages, 0 → 0 reconciliations), so there is nothing to charge and the CTO removed pricing from scope." That measurement was subsequently shown to be **the signature of `DROPCAP_TEXT_OVERLAP`, not evidence of no cost** — PDFKit overlapped the lines instead of growing them, while Word grows 3 → 4 pages on the same content. Pricing was consequently **reinstated** (the reversal in `docs/DECISIONS.md`) and now ships on `main` (`LayoutEngine.priceDropCapParagraph`): a drop cap *does* carry a real, charged height cost. The atomicity debt stands anyway — not because the glyph is free, but because an unsplittable paragraph can overflow no matter how precisely its height is charged.

  **The CTO's condition: the debt may stay uncorrected but may not stay unmeasured.** The obligation is a test proving the property — a drop-cap paragraph too long for the page remainder yields a *measured reconciliation*, never a silent overflow — the letter of ADR-0051: loud, not silent. **This test is still owed:** the overlap fix's own test (`dropCapOverlap.test.ts`) asserts `unplannedPageBreaks === 0` on a page that *fits*, which is the pricing property, not this atomicity property; no test yet forces a drop-cap paragraph to overflow a page remainder. It is a candidate for a future splitting review alongside its cousin `LIST_SPLITTING_ACROSS_PAGES`, not a hidden gap.

## 🐛 KNOWN ISSUES

- Legacy `/api/upload` route removed (ADR-0021, PR #5, 2026-07-17) — `POST /api/manuscripts/import` is now the only import route.
- `backend/uploads/` no longer tracked going forward (`.gitignore` + `git rm --cached`), but still present in past commit history — decided to keep as-is, no purge (ADR-0021).
- Mammoth (DOCX import) silently drops underline formatting by default — documented dependency limitation, not a pipeline bug (ADR-0025, Sprint 4). Verified workaround exists, not applied; requires a mammoth-level `styleMap` fix, scoped for Import Fidelity.
- PDFKit has no native primitive for superscript/subscript/small-caps — `PDFRenderer` documents these `TypeRun` flags as unrendered (DOCX/EPUB render them correctly).
- **DOCX footnotes are rendered inline as `[N]` paragraphs, not native Word footnote references/definitions** (surfaced by chantier B / `verify-publication-quality`, CTO-classified 2026-07-21). A live rendering choice on `main` (`DOCXRenderer.renderBlock` case `'footnote'`), not a defect B fixes. `verify-publication-quality` §4.3 accepts the inline-`[N]` proxy (a `[N]` matches a present definition) with this limitation disclosed; it does **not** verify true native Word footnotes. **Known fidelity gap** — candidate for a future review if a publisher ever requires real Word footnotes. Also a **coverage gap**: zero `footnoteReference` in any fixture repo-wide (measured), and no fixture is fabricated to fill it.
- **DOCX hyperlinks: external URLs only; internal anchors (cross-references) are unimplemented** (surfaced by chantier B, CTO-confirmed out of scope 2026-07-21). `DOCXRenderer` emits `ExternalHyperlink` for `TypeRun.linkUrl`; there is no bookmark/anchor machinery. `verify-publication-quality` §4.7 verifies external targets and discloses internal anchors as out of scope. **Coverage gap**: zero `w:hyperlink` in any fixture repo-wide (measured); no fixture fabricated.
- ~~Strikethrough silently downgraded to plain text on import; inter-run whitespace silently dropped (word-jamming); `ASTBuilder.convertInlines()` dropped plain text + silently mislabeled unknown inline types as bold~~ — found and **fixed** in Sprint 4 commit 10 as an explicit scope exception (ADR-0026), not left open.
- `Chapter.openingPageStyle`/`startPageNumber`/`Book.frontMatter.toc.generateAutomatically` have no DOCX-native signal `ASTBuilder` can set from real content (Sprint 6, ADR-0031) — same category as the `isbn`/`description`/`coverImage` gap below. Fully implemented, real-file verified via direct pipeline composition; not reachable through a real DOCX upload today. Scoped for a future Import Fidelity sprint.
- DOCX `RunningHead.content:'chapterTitle'` and `Chapter.startPageNumber` have no real per-chapter effect in `.docx` output (Sprint 6) — `DOCXRenderer` builds one Word section for the whole document; true per-chapter behavior needs splitting into multiple sections, not built this sprint. Doesn't affect `ClassicTheme`, whose `'bookTitle'` content is constant document-wide regardless.
